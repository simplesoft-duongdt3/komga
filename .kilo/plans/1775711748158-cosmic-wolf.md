# Plan: Fix PostgreSQL Temporary Table and Collation Issues

## Overview
Komga has two main PostgreSQL-related issues:
1. **Temporary table "does not exist" errors** due to connection pooling and transaction boundaries
2. **Collation "und-u-ks-level2" not found** in PostgreSQL Alpine images

Both issues prevent background tasks (RebuildIndex, ScanLibrary) and some API operations from functioning correctly.

## Root Cause Analysis

### 1. Temporary Table Issues
**Problem**: PostgreSQL temporary tables are session-scoped (visible only within same database connection). With connection pooling and no transaction boundaries, different queries may use different connections, making temp tables invisible.

**Code Patterns**:
- **Working**: Delete operations with `@Transactional` annotations use `dslRW.withTempTable()` and maintain same connection
- **Broken**: DTO DAO query methods without transactions (e.g., `BookDtoDao.findAll()`, `SeriesDtoDao.fetchAndMap()`) create temp tables on `dslRO` connection but subsequent queries may use different connections

**Connection Management**: The `SplitDslDaoBase` logic returns `dslRW` when a read-write transaction is active, otherwise returns separate `dslRO` connection. PostgreSQL uses single connection pool (no separate read/write pools), but still has separate `DSLContext` beans.

### 2. Collation Issues
**Problem**: PostgreSQL Alpine images have limited ICU support. The hardcoded collation `"und-u-ks-level2"` (used for case-insensitive, accent-insensitive Unicode sorting) doesn't exist in Alpine's minimal ICU installation.

**Location**: `PostgresUdfProvider.kt` line 22: `this.collate("und-u-ks-level2")`

**Impact**: All sorted queries fail, affecting:
- Series/book listing with title sorting
- Author/genre/tag/publisher listings
- Search result ordering

## Solution Strategy

### Phase 1: Fix Temporary Table Connection Issues

#### 1.1 Add Transaction Boundaries to DTO DAO Methods
**Approach**: Add `@Transactional(readOnly = true)` to query methods using temp tables:
- `BookDtoDao.findAll()` and related query methods
- `SeriesDtoDao.findAll()` methods  
- `BookDtoDao.fetchAndMap()` (called by `findAll()`)
- `SeriesDtoDao.fetchAndMap()` (called by `findAll()`)

**Rationale**: Ensures all temp table operations within method use same connection. `readOnly = true` appropriate for query operations.

**Files to modify**:
- `komga/src/main/kotlin/org/gotson/komga/infrastructure/jooq/main/BookDtoDao.kt`
- `komga/src/main/kotlin/org/gotson/komga/infrastructure/jooq/main/SeriesDtoDao.kt`

**Example change**:
```kotlin
@Transactional(readOnly = true)
override fun findAll(
  userId: String,
  filter: BookFilter?,
  pageable: Pageable,
  restrictions: ContentRestrictions,
): Page<BookDto> {
  // existing implementation
}
```

**Transaction Propagation**: Use default `REQUIRED` propagation. If method is called within an existing transaction, it will join that transaction. The `readOnly` flag may be ignored when joining an existing read-write transaction, which is acceptable.

**Note**: Need to check if methods are already in `@Transactional` service layer. If so, adding `@Transactional` at DAO level may be redundant but harmless. However, we must ensure the DAO methods have transaction boundaries when called directly (e.g., from background tasks).

**Check for similar patterns** in:

**Action**: Add `@Transactional(readOnly = true)` where appropriate for query methods.

#### 1.3 Enhance TempTable for PostgreSQL Compatibility
**Improvements to consider**:

1. **Transaction-scoped temp tables**: Use `ON COMMIT DROP` clause in PostgreSQL:
   ```sql
   CREATE TEMPORARY TABLE $name (STRING varchar NOT NULL) ON COMMIT DROP
   ```
   This automatically drops the table at end of transaction, providing cleaner lifecycle management.

2. **Connection validation**: Ensure all operations use same connection by storing `Connection` reference alongside `DSLContext`.

3. **Batch size optimization**: Review batch size defaults for PostgreSQL (currently 1000). PostgreSQL may have different optimal batch sizes.

**Evaluation**: `ON COMMIT DROP` is elegant but requires all temp table usage to be within transactions (which we're ensuring anyway). Connection validation provides defensive programming but adds complexity.

### Phase 2: Fix PostgreSQL Collation Issues

#### 2.1 Update Docker PostgreSQL Image
**Immediate fix**: Change from Alpine to standard PostgreSQL image in `docker-compose.yml`:
```yaml
image: postgres:15  # instead of postgres:15-alpine
```

**Pros**: Simple, ensures full ICU support
**Cons**: Larger image size, not a code fix

#### 2.2 Implement ICU Collation Detection and Fallback
**Robust solution**: Modify `PostgresUdfProvider` to:
1. Detect ICU support at startup (in `initializeConnection`)
2. Fall back to alternative collation if `"und-u-ks-level2"` unavailable
3. Log warning when using fallback

**Implementation steps**:
- Add collation detection in `initializeConnection` method
- Cache collation availability in a thread-safe manner
- Use `COLLATE "C"` (binary) or PostgreSQL's `citext` extension as fallback
- Make collation name configurable via properties

**Files to modify**:
- `komga/src/main/kotlin/org/gotson/komga/infrastructure/datasource/PostgresUdfProvider.kt`
- Add configuration property in `KomgaProperties`

**Example implementation**:
```kotlin
class PostgresUdfProvider : DatabaseUdfProvider {
  private var icuCollationAvailable: Boolean? = null
  private val lock = Any()
  
  override fun Field<String>.collateUnicode3(): Field<String> {
    val collationName = if (isIcuCollationAvailable()) "und-u-ks-level2" else "C"
    return this.collate(collationName)
  }
  
  private fun isIcuCollationAvailable(): Boolean {
    if (icuCollationAvailable != null) return icuCollationAvailable!!
    
    synchronized(lock) {
      // Check if collation exists in pg_collation
      // This check could be done once per connection in initializeConnection
      return icuCollationAvailable ?: false
    }
  }
  
  override fun initializeConnection(connection: Any) {
    // Existing unaccent extension check...
    
    // Check ICU collation availability
    val pgConnection = connection as Connection
    try {
      val checkCollationSQL = """
        SELECT 1 FROM pg_collation 
        WHERE collname = 'und-u-ks-level2' 
        AND collencoding = -1  -- Works for any encoding
      """
      val rs = pgConnection.createStatement().executeQuery(checkCollationSQL)
      icuCollationAvailable = rs.next()
      log.info { "ICU collation 'und-u-ks-level2' available: $icuCollationAvailable" }
    } catch (e: Exception) {
      log.error(e) { "Failed to check ICU collation availability" }
      icuCollationAvailable = false
    }
  }
}
```

#### 2.3 Add Collation Creation Script
**Alternative**: Create the ICU collation in PostgreSQL initialization if missing.

**Script addition** to `docker/postgres/init.sql`:
```sql
-- Attempt to create ICU collation if supported
DO $$
BEGIN
  IF NOT EXISTS (
    SELECT 1 FROM pg_collation WHERE collname = 'und-u-ks-level2'
  ) THEN
    -- Create collation if ICU available
    CREATE COLLATION "und-u-ks-level2" (LOCALE = 'und-u-ks-level2');
  END IF;
EXCEPTION WHEN others THEN
  RAISE NOTICE 'Could not create und-u-ks-level2 collation: %', SQLERRM;
END $$;
```

**Limitation**: May not work on Alpine if ICU data missing.

## Risks and Mitigation

### Risk 1: Transaction Overhead
Adding `@Transactional` to query methods may introduce overhead.

**Mitigation**: 
- Use `readOnly = true` to allow optimization
- Test performance impact
- Consider transaction propagation (`REQUIRED` vs `SUPPORTS`)

### Risk 2: Collation Behavior Changes
Different collations may produce different sort orders.

**Mitigation**:
- Document behavior differences
- Maintain backward compatibility where possible
- Allow configuration override

### Risk 3: PostgreSQL Version Compatibility
Some features may not work across all PostgreSQL versions.

**Mitigation**:
- Test with multiple PostgreSQL versions (15, 16)
- Use feature detection rather than version checks
- Provide clear documentation
