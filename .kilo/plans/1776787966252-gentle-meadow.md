# Komga Rust Migration Plan

## Current Architecture Analysis

### Technology Stack
- **Backend**: Spring Boot (Kotlin) with jOOQ (not JPA!)
- **Database**: PostgreSQL + SQLite (Flyway migrations)
- **Caching**: Caffeine (in-memory) - one source of RAM bloat
- **Search**: Lucene (loaded in memory)
- **Session**: Spring Session with Caffeine - another RAM source

### Domain Models (Pure Kotlin data classes - easy to port)
- Book, Series, Library, ReadList, ReadProgress
- Media, MediaFile (for ebook parsing)
- User, ApiKey, AuthenticationActivity

### Key Infrastructure
- jOOQ-based DAOs (great SQL reference for Rust)
- Task system for async processing
- Image processing (thumbnail generation)
- Metadata extraction (PDF, EPUB, CBZ/CBR)

---

## Recommendations for komga-rust

### 1. Database Layer
- **Drop SQLite completely** - simplifies migration
- Use **SQLx** or **diesel** + **postgresql** for PostgreSQL only
- Consider using the **existing Flyway migrations** (convert to raw SQL or Rust migrations)
- Keep the schema, just port the jOOQ DAOs to Rust

### 2. Redis Integration (for RAM reduction)
- Replace Caffeine session store with Redis
- Cache strategy:
  - `TransientBookCache` → Redis (hot data)
  - Thumbnail cache → Redis or file-based (LRU)
  - User sessions → Redis
  - Lucene search indexes → consider **tantivy** (Rust native) instead of keeping in-memory

### 3. Rust Web Framework
- **Axum** or **Actix-web** - both mature, Axum more ergonomic with tokio
- Consider **Salvo** for better HTTP/2 support if needed

### 4. Key Libraries to Consider

| Purpose | Java (current) | Rust suggestion |
|---------|---------------|----------------|
| Database | jOOQ | SQLx or Diesel |
| Search | Lucene | **Tantivy** (Rust native, lower RAM) |
| Image proc | Thumbnailator, PDFBox | **image**, **pdf** crates |
| EPUB | Epublib | **epub** crate |
| Archive | Junrar, Commons Compress | **zip**, **rar** crates |
| Validation | Apache Commons Validator | regex/ custom |
| Serialization | Jackson | **serde** + **serde_json** |
| Auth | Spring Security | **axum-auth** or custom |
| OAuth2 | Spring Security OAuth2 | **oauth2** crate |
| Caching | Caffeine | **redis** + local LRU |

### 5. RAM Optimization Strategy
1. **Lazy loading** - don't load Lucene indexes at startup
2. **Streaming** - process large files in chunks, not memory
3. **Redis caching** - externalize cache from process memory
4. **Tantivy over Lucene** - more memory-efficient, mmap-based

### 6. Migration Approach
- **Option A**: Rewrite from scratch (clean slate, longer)
- **Option B**: Incremental port (keep Java for heavy processing, Rust for API)
- **Recommended**: Option A but reuse:
  - Flyway SQL migrations (just copy)
  - Domain model structure (rewrite in Rust)
  - API contract (maintain compatibility)

### 7. Priority Order
1. Database schema + migrations (PostgreSQL only)
2. Core domain models (Book, Series, Library)
3. REST API layer (maintain compatibility)
4. Redis integration for session/cache
5. Background tasks (async processing)
6. Search (Tantivy)
7. File processing (image, ebook parsing)

## API Compatibility Strategy

- Use existing OpenAPI spec: `komga/docs/openapi.json` (14k+ lines)
- Generate Rust DTOs using `openapi-generator` from the JSON spec
- Write contract tests that hit both Java and Rust backends
- Ensure response DTOs match exactly (JSON field names, ordering)
- Test with existing komga-webui against Rust backend

---

## Key Decisions

- **Task queue**: PostgreSQL (as-is, reuse existing task schema)
- **OpenAPI**: Generate from Rust (use `utoipa`)
- **Existing OpenAPI spec**: `komga/docs/openapi.json` - use this to generate Rust DTOs with `openapi-generator`
