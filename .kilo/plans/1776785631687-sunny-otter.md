# Issues Analysis & Solutions

## Issue 1: ScanLibrary Creates Duplicate Series with Same Folder Path

### Problem Summary
When running `ScanLibrary`, duplicate series entries can be created for the same folder path.

### Root Causes

1. **No Database-Level UNIQUE Constraint**
   - The `SERIES` table has NO unique constraint on `URL` column or `(LIBRARY_ID, URL)` combination
   - File: `komga/src/flyway/resources/db/migration/postgresql/V001__initial_migration.sql:61-74`
   - Only has PRIMARY KEY on `ID` and FOREIGN KEY on `LIBRARY_ID`

2. **Race Condition During Concurrent Scans**
   - If two `ScanLibrary` tasks run simultaneously for the same library, both can load existing series, find no match for a new URL, and both attempt to create it
   - File: `komga/src/main/kotlin/org/gotson/komga/domain/service/LibraryContentLifecycle.kt:126-163`
   - Series are loaded into memory, then checked via `existingActiveSeriesByUrl[newSeries.url]` before creation

3. **Soft-Deleted Series Ignored**
   - When checking for existing series, only non-deleted series are considered:
   ```kotlin
   val existingActiveSeriesByUrl =
     existingSeriesById.values
       .filter { it.deletedDate == null }  // <-- Only non-deleted
   ```
   - A new series with the same URL as a soft-deleted one will be created as a duplicate

4. **Plain INSERT with No Conflict Handling**
   - File: `komga/src/main/kotlin/org/gotson/komga/infrastructure/jooq/main/SeriesDao.kt:166-177`
   - Uses plain `INSERT` with no `ON CONFLICT` clause

### Solution

#### Step 1: Add Database-Level UNIQUE Constraint
Create a new Flyway migration to add a unique constraint:

```sql
-- New migration file: V{version}__add_unique_constraint_series_url.sql
ALTER TABLE "SERIES" ADD CONSTRAINT "UQ_SERIES_LIBRARY_URL" UNIQUE ("LIBRARY_ID", "URL");
```

#### Step 2: Add Conflict Handling in SeriesDao
Modify `SeriesDao.insert()` to handle conflicts:

```kotlin
// komga/src/main/kotlin/org/gotson/komga/infrastructure/jooq/main/SeriesDao.kt
```

## Local Assets/Thumbnails Documentation

### Storage Architecture

Komga uses a **hybrid storage approach** for thumbnails:

| Storage Type | Location | Used For |
|--------------|----------|----------|
| **Database BLOB** | `THUMBNAIL_BOOK.THUMBNAIL` column | Generated thumbnails, User-uploaded thumbnails |
| **File System URL** | `THUMBNAIL_BOOK.URL` column | Sidecar (local artwork) thumbnails |

### Database Schema

**Table:** `THUMBNAIL_BOOK`
- File: `komga/src/flyway/resources/db/migration/postgresql/V001__initial_migration.sql:179-194`

```sql
CREATE TABLE "THUMBNAIL_BOOK"
(
    "ID"                 varchar   NOT NULL PRIMARY KEY,
    "THUMBNAIL"          bytea     NULL,        -- BLOB for generated/uploaded thumbnails
    "URL"                varchar   NULL,        -- File URL for sidecar thumbnails
    "SELECTED"           boolean   NOT NULL DEFAULT false,
    "TYPE"               varchar   NOT NULL,    -- GENERATED, SIDECAR, USER_UPLOADED
    "WIDTH"              integer   NOT NULL DEFAULT 0,
    "HEIGHT"             integer   NOT NULL DEFAULT 0,
    "MEDIA_TYPE"         varchar   NOT NULL DEFAULT '',
    "FILE_SIZE"          bigint    NOT NULL DEFAULT 0,
    "BOOK_ID"            varchar   NOT NULL,
    FOREIGN KEY ("BOOK_ID") REFERENCES "BOOK" ("ID")
);
```

### Thumbnail Types

1. **GENERATED** - Auto-extracted from book content (first page/cover)
2. **SIDECAR** - Local artwork file stored alongside book/series
3. **USER_UPLOADED** - Manually uploaded by user

### Folder Structure for Local Assets (Sidecars)

**Book Sidecars** - Stored in same folder as book file:
- Pattern: `<bookname>(-<number>).<ext>`
- Examples: `Book 1.jpg`, `Book 1-1.jpg`, `Book 1-2.png`

**Series Sidecars** - Stored in series folder:
- File names: `cover`, `default`, `folder`, `poster`, `series`
- Examples: `cover.jpg`, `poster.png`, `folder.webp`

**Supported Extensions:** `png`, `jpeg`, `jpg`, `tbn`, `webp`, `gif`

**Example Folder Structure:**
```
/Library Root/
  └── Series Name/
      ├── cover.jpg              ← Series thumbnail (sidecar)
      ├── poster.png             ← Series thumbnail (sidecar)
      ├── Book 1.cbz
      ├── Book 1.jpg             ← Book thumbnail (sidecar)
      ├── Book 1-1.jpg           ← Additional book thumbnail
      ├── Book 1-2.png           ← Additional book thumbnail
      └── Book 2.cbz
```

### How Thumbnails Are Read

**Reading Logic** (BookLifecycle.kt:239-289):
```kotlin
fun getThumbnailBytes(
  bookId: String,
  resizeTo: Int? = null,
): TypedBytes? {
  getThumbnail(bookId)?.let {
    val thumbnailBytes =
      when {
        it.thumbnail != null -> it.thumbnail          // From DB BLOB
        it.url != null -> File(it.url.toURI()).readBytes()  // From file system
        else -> return null
      }
    
    if (resizeTo != null) {
      return TypedBytes(
        imageConverter.resizeImageToByteArray(thumbnailBytes, resizeTargetFormat, resizeTo),
        resizeTargetFormat.mediaType,
      )
    }
    
    return TypedBytes(thumbnailBytes, it.mediaType)
  }
  return null
}
```

### Configuration

**Library Settings** (Library.kt:9-60):
- `importLocalArtwork: Boolean` - Enable/disable local artwork import (default: true)
- `seriesCover: SeriesCover` - Which book cover to use for series thumbnail

**Global Settings:**
- `thumbnailSize: ThumbnailSize` - DEFAULT(300), MEDIUM(600), LARGE(900), XLARGE(1200)
- Configurable at runtime via settings

### Caching

- **No application-level image caching** - Thumbnails read from DB/file on each request
- **HTTP cache headers** used for client-side caching
- **Housekeeping** removes invalid sidecar references when files are deleted

### Root Causes

1. **Single-Threaded Task Processing (Default)**
   - File: `komga/src/main/kotlin/org/gotson/komga/infrastructure/configuration/KomgaSettingsProvider.kt:60-66`
   - Default `taskPoolSize` is **1**, meaning tasks run sequentially
   - File: `komga/src/main/kotlin/org/gotson/komga/application/tasks/TaskProcessor.kt:15-72`

2. **CPU-Intensive Image Processing**
   - File: `komga/src/main/kotlin/org/gotson/komga/infrastructure/image/ImageConverter.kt:93-138`
   - Full image decode → resize → encode cycle for each thumbnail
   - Uses `thumbnailator` library for high-quality resizing

3. **PDF Rendering is Slow**
   - File: `komga/src/main/kotlin/org/gotson/komga/infrastructure/mediacontainer/pdf/PdfExtractor.kt:39-53`
   - PDF pages are rendered to images using PDFBox
   - Large/high-resolution PDFs require significant CPU and memory

4. **Book Analysis Pre-requisite**
   - File: `komga/src/main/kotlin/org/gotson/komga/domain/service/BookLifecycle.kt:78-103`
   - Before thumbnail generation, book must be analyzed (opening archives, detecting media types, reading all page dimensions)

5. **No In-Memory Thumbnail Cache**
   - Thumbnails are read from database on each request
   - No caching layer for frequently accessed thumbnails

### Solution

#### Step 1: Increase Task Pool Size
Allow configurable parallel task execution:

```kotlin
// komga/src/main/kotlin/org/gotson/komga/infrastructure/configuration/KomgaSettingsProvider.kt
// Increase default pool size based on available processors
var taskPoolSize: Int =
  serverSettingsDao.getSettingByKey(Settings.TASK_POOL_SIZE.name, Int::class.java) 
    ?: Runtime.getRuntime().availableProcessors().coerceAtMost(4)
```

#### Step 2: Add Thumbnail Generation Cache
Cache recently generated thumbnails to avoid redundant processing:

```kotlin
// New file: komga/src/main/kotlin/org/gotson/komga/infrastructure/cache/ThumbnailCache.kt
class ThumbnailCache(
  private val komgaSettingsProvider: KomgaSettingsProvider,
) {
  private val cache =
    Caffeine.newBuilder()
      .maximumSize(100)  // Configurable
      .expireAfterWrite(24, TimeUnit.HOURS)
      .build<String, ByteArray>()  // bookId -> thumbnail bytes

  fun getThumbnail(bookId: String): ByteArray? = cache.getIfPresent(bookId)
  
  fun putThumbnail(bookId: String, thumbnail: ByteArray) {
    cache.put(bookId, thumbnail)
  }
}
```

#### Step 3: Optimize Image Processing Pipeline
Add early-exit conditions and optimize resizing:

```kotlin
// komga/src/main/kotlin/org/gotson/komga/infrastructure/image/ImageConverter.kt
fun resizeImageToByteArray(
  imageBytes: ByteArray,
  format: ImageType,
  size: Int,
): ByteArray {
  // Check if already cached
  val cacheKey = generateCacheKey(imageBytes, format, size)
  thumbnailCache.get(cacheKey)?.let { return it }
  
  // Skip resize if image is already smaller than target
  val dimension = imageAnalyzer.getDimension(imageBytes.inputStream())
    ?: return imageBytes
  val longestEdge = max(dimension.height, dimension.width)
  if (longestEdge <= size) return imageBytes
  
  // Use faster resizing for large batches
  val result = ByteArrayOutputStream().use { out ->
    Thumbnails.of(imageBytes.inputStream())
      .size(size, size)
      .imageType(BufferedImage.TYPE_INT_ARGB)
      .outputFormat(format.imageIOFormat)
      .toOutputStream(out)
    out.toByteArray()
  }
  
  thumbnailCache.put(cacheKey, result)
  return result
}
```

#### Step 6: Add Thumbnail Regeneration Throttling
Prevent overwhelming the system when regenerating many thumbnails:

```kotlin
// komga/src/main/kotlin/org/gotson/komga/application/tasks/TaskEmitter.kt
fun generateBookThumbnailsThrottled(
  bookIds: Collection<String>,
  priority: Int = DEFAULT_PRIORITY,
  delayMs: Long = 100,  // Configurable delay between tasks
) {
  bookIds.forEachIndexed { index, bookId ->
    Thread.sleep(index * delayMs)  // Stagger task submission
    generateBookThumbnail(bookId, priority)
  }
}
```
